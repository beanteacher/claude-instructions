# 에러 처리 (Iron Rule)

- **경계에서는 빠르게 실패하고, 부작용에 대해서는 안전하게 실패한다.**
  - 컨트롤러/서비스 진입부에서 선행 조건을 즉시 검증하고 비즈니스 예외로 던진다.
  - 외부 시스템 연동·파일 쓰기 같은 부수효과는 afterCommit 리스너 또는 별도 보상 로직을 통해 **실패 시 롤백 또는 재시도 가능한 상태**로 둔다.
- 내부 예외를 사용자 안전 에러 응답으로 변환한다.
  - 비즈니스 오류 → `AppException` (또는 프로젝트 표준 예외) → `GlobalExceptionHandler` → `ErrorResponse` (JSON) 로 통일.
  - 새 오류 케이스가 필요하면 `ErrorCode` enum 에 항목을 먼저 추가하고, 메시지·HttpStatus·코드를 한 곳에서 관리한다.
  - 서비스/컨트롤러 안에서 try/catch 로 응답을 직접 만들지 않는다. 인프라 경계(외부 SDK 호출 실패 등)에서만 래핑 후 재던진다.
- **에러를 조용히 삼키지 않는다.**
  - `catch (Exception e) { /* ignore */ }` 패턴 금지.
  - 의도적으로 무시해야 한다면 **왜 무시해도 되는지 한 줄 주석**으로 근거를 남기고, `log.warn(...)` 은 반드시 찍는다.
- 가능한 경우 **상관 식별자(request ID, resource id, 사용자 ID 등)가 포함된 실행 가능한 로그**를 생성한다.
  - `@Slf4j` + 파라미터 로깅을 기본으로, MDC 필요 시 로깅 AOP 와 연계.
  - 민감 정보(토큰, 패스워드, OTP)는 마스킹 처리한다.
- **변경된 비즈니스 로직에 대해 최소 하나의 실패 경로 테스트를 포함한다.** 성공 경로만 테스트하는 PR 은 반려 대상이다.
- 예외 계층 재정의 금지. 도메인마다 `XxxNotFoundException`, `XxxDuplicateException` 을 새로 만들지 않는다 — 공통 `AppException(ErrorCode.X)` 로 통일.
- 공통 리포지토리에 `findByIdOrThrow(id)` 가 있으면 `findById().orElseThrow(...)` 를 중복 작성하지 않는다.
- **afterCommit 리스너의 예외**는 해당 트랜잭션을 되돌리지 않는다. 부수효과 실패 시 적절한 보상 로직 또는 알림 경로로 에스컬레이션한다.
