# 완전성 원칙 (Iron Rule)

- **증분 비용이 낮을 때는 완전하고 검증 가능한 구현을 선호한다.**
  - 한 엔티티를 추가하면서 repository/service/controller/test 까지 함께 쌓는다.
  - DDL 을 바꾸면서 스키마 소스 파일을 같이 수정한다.
  - 보안 경로를 건드리면 `SecurityConfig` + 테스트 + 공개 경로 주석까지 같이 업데이트한다.
- **엣지 케이스와 테스트가 간단한 경우 "80–90% 완료"를 미완성으로 취급한다.**
  - "일단 happy path 만 머지하고 edge case 는 나중에" 스타일 지양.
  - null 필드, 예약어 컬럼, 중복 제약 위반 같은 **전형적 엣지 케이스**는 작성 시점에 함께 고려한다.
- 다음을 구분한다:
  - **호수(lake)** — 지금 완료할 수 있는 한정된 작업. 예: "삭제 가드 조건 하나 추가". → 한 번에 끝까지 구현한다.
  - **바다(ocean)** — 명시적 범위 결정이 필요한 무한한 재작성/마이그레이션. 예: "DB 엔진 전환", "전체 도메인을 Modulith 로 재편". → 범위를 쪼개 단계화된 체크리스트로 정리하고, 매 단계마다 독립적으로 머지 가능한 상태로 끝낸다.
- **리뷰 시 회피 가능한 편법과 누락된 검증을 지적한다.**
  - `TODO` / `FIXME` 를 남기면서 테스트·문서·DDL 중 하나를 건너뛴 코드는 리뷰에서 반려.
  - `@Disabled`, `Assumptions.assumeFalse(...)` 로 끈 테스트는 **복구 계획**을 commit 메시지/PR body 에 명시하지 않으면 반려.
  - "일단 log 만 남기고 진짜 동작은 다음 PR" 같은 스텁은 피한다 — 만들 거면 끝까지, 아니면 분리된 TODO 이슈로 추적.

## 새 기능 완료 전 체크리스트

새 기능을 완료로 선언하기 전에 아래 항목을 모두 확인한다:

```
[ ] 엔티티·DTO·레포지토리·서비스·컨트롤러가 서로 정합하게 연결되는가?
[ ] DDL(스키마 파일)이 함께 수정되었는가?
[ ] SecurityConfig 의 permitAll 목록에 경로를 추가/제거해야 하는가?
[ ] 단위 + 슬라이스 테스트가 성공·실패 양쪽 경로를 커버하는가?
[ ] 로그에 민감 정보가 노출되지 않는가?
[ ] application-{local,development,production}.yml 에 환경별 값이 모두 있는가?
[ ] CI/CD 파이프라인에 영향받는 구성 변경이 있는가?
```
