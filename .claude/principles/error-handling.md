# 에러 처리 (Iron Rule)

- **경계에서는 빠르게 실패하고, 부수효과/외부 호출에 대해서는 안전하게 실패한다.**
  - 진입부에서 선행 조건을 즉시 검증하고 도메인 예외로 던진다.
  - 외부 시스템 연동·파일 쓰기 같은 부수효과는 **실패 시 롤백 또는 재시도 가능한 상태**로 둔다. 부분 실패는 격리한다.
- **내부 예외를 사용자 안전 에러 응답으로 변환한다.**
  - 비즈니스 오류 → 도메인 예외 → 공통 핸들러 → 표준 에러 응답 으로 통일한다.
  - 새 오류 케이스가 필요하면 공통 에러 정의에 항목을 먼저 추가하고, 메시지·상태·코드를 한 곳에서 관리한다.
  - 서비스/핸들러 안에서 try/catch 로 에러 응답을 직접 만들지 않는다. 인프라 경계(외부 SDK 호출 실패 등)에서만 래핑 후 재던진다.
- **예외 계층을 재정의하지 않는다.** 도메인마다 `XxxNotFound...` 류를 새로 만들지 않고 공통 에러 계층으로 통일한다.
- **에러를 조용히 삼키지 않는다.**
  - `catch (e) { /* ignore */ }` 패턴 금지.
  - 의도적으로 무시해야 한다면 **왜 무시해도 되는지 한 줄 주석**으로 근거를 남기고, 경고 로깅(`warn`)은 반드시 남긴다.
- **상관 식별자가 포함된 실행 가능한 로그를 남긴다.** (request id, resource id, 사용자 id 등) 민감 정보(토큰·패스워드·OTP)는 마스킹한다.
- **변경된 비즈니스 로직에 대해 최소 하나의 실패 경로 테스트를 포함한다.** 성공 경로만 테스트하는 PR 은 반려 대상이다.

## 체크리스트

```
[ ] 새 오류는 공통 에러 정의에 항목으로 추가했는가?
[ ] 핸들러/서비스에서 직접 에러 응답을 만들지 않았는가?
[ ] 외부 호출에 재시도 또는 명시적 예외 처리가 있는가?
[ ] 로그에 상관 식별자가 포함되는가? (민감 정보는 마스킹)
[ ] 에러를 조용히 삼키는 패턴이 없는가? (있다면 WHY 주석 + 경고 로깅)
[ ] 실패 경로 테스트가 최소 하나 이상 있는가?
```
