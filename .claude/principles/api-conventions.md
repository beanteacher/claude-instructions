# API 규약 (Iron Rule)

- **경계에서 요청 입력을 검증한다. 클라이언트 페이로드를 절대 신뢰하지 않는다.**
  - 모든 요청 DTO 는 경계에서 검증 규칙(필수/길이/범위/패턴 등)과 함께 선언한다.
  - 검증 실패는 공통 핸들러가 표준 검증 오류로 변환하도록 하고, 각 핸들러에서 개별 처리하지 않는다.
  - 식별자는 경로 파라미터로 받되 **서비스 레벨에서 소유권·권한을 재검증**한다 (URL 에 있다는 이유로 통과시키지 않는다).
- **명시적 에러 코드가 포함된 안정적인 응답 형태를 반환한다.**
  - 성공 응답은 도메인 응답 타입으로 반환한다.
  - 실패 응답은 공통 예외 핸들러가 일괄 변환한다. 핸들러/라우터에서 직접 에러 응답 body 를 구성하지 않는다.
  - 새 오류는 공통 에러 정의(코드 + 메시지 + 상태)에 항목을 추가한 뒤 도메인 예외로 던진다.
- **핸들러는 가볍게 유지한다. 비즈니스 로직은 서비스 레이어로 이동한다.**
  - 핸들러 메서드는 기본적으로 3~5줄. 서비스 호출과 응답 래핑 외의 로직을 두지 않는다.
  - 분기가 3개 이상 생기면 서비스 내부 private helper 로 분리한다.

---

## URL 규칙

- 버전 prefix(`/api/vN/...`) 사용 여부는 프로젝트 관행을 따른다. 버전 분리가 필요 없으면 도메인 경로로 직접 시작할 수 있다. 버전 정책은 프로젝트 시작 시 결정하고 `CLAUDE.md` 에 명시한다.
- **공개 경로는 반드시 명시적 allowlist 에 등록**한다. 기본은 항상 인증 요구.
- HTTP method 시맨틱을 지킨다.
  ```
  GET    조회
  POST   생성
  PUT    전체 수정
  PATCH  부분 수정
  DELETE 삭제
  ```
- 상태 변경을 "POST /xxx/do-something" 같은 동사형 경로로 만들지 않는다. 리소스 중심으로 `PUT /xxx/{id}/state` 또는 전용 하위 리소스로 모델링한다.

---

## 응답 형태

- 단건: 도메인 응답 타입.
- 목록(페이지 없음): 목록 래퍼 타입.
- 목록(페이지): 페이지 래퍼 타입.
- 단순 성공 확인: 표준 성공 응답.
- 실패: 공통 예외 핸들러가 표준 에러 형태로 일괄 변환한다.

---

## Pagination / Sorting

- 프로젝트 표준 페이지네이션 방식(페이지/사이즈 또는 limit/offset)을 따르고, 기본값을 한 곳에서 관리한다. 하드코딩하지 않는다.

---

## 공개 경로 / Webhook

- 공개 경로(webhook 등)는 인증을 면제하더라도 **애플리케이션 단에서 자체 검증을 수행한다** (서명 헤더 / 공유 토큰 검증 등).
- 공격자가 임의 payload 로 호출할 수 있다는 전제로 작성한다.
