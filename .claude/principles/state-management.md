# 상태 관리 원칙 (Iron Rule)

> 적용 범위: 프론트엔드 프로젝트

> 상태를 여러 레이어로 명확히 분리한다.
> "어디에 상태를 두어야 하는가"에 대한 답은 아래 의사결정 트리를 먼저 참조한다.

---

## 의사결정 트리

```
이 상태는 어디에 두는가?
│
├─ 백엔드 API 에서 가져오는 데이터인가?
│  └─ YES → 서버 상태 캐싱 라이브러리 (서버 상태)
│
├─ 사용자 인증 정보·액세스 토큰인가?
│  └─ YES → 클라이언트 메모리 스토어 (인메모리 — 영구 저장소 금지)
│
├─ URL 에 공유되어야 하는 필터·페이지·정렬인가?
│  └─ YES → URL 상태 (쿼리스트링 동기화 훅)
│
├─ 폼 입력값인가?
│  └─ YES → 폼 상태 라이브러리 + 스키마 검증
│
└─ 그 외 (모달 open/close, 선택된 탭 등 UI 지역 상태)
   └─ 컴포넌트 지역 상태
```

---

## 1. 서버 상태

- 모든 백엔드 데이터 페칭·캐싱·동기화는 서버 상태 캐싱 라이브러리를 사용한다. 컴포넌트 안에서 직접 fetch 후 지역 상태에 보관하지 않는다.
- 전역 기본 옵션(staleTime, retry 등)은 한 곳에서 설정하고, 개별 훅에서 동일 옵션을 중복 선언하지 않는다.
- 쿼리 키는 키 팩토리로 한 곳에서 관리한다. 문자열 배열을 호출 시마다 인라인으로 하드코딩하지 않는다.
- 변경(mutation) 성공 후에는 관련 쿼리를 무효화(invalidate)해 캐시를 동기화한다.

---

## 2. 클라이언트 상태

새 전역 스토어는 아래 조건을 모두 충족할 때만 만든다.
1. 여러 컴포넌트 트리에서 공유가 필요하다.
2. URL 에 넣을 수 없는 상태다 (민감 정보 또는 세션 한정 상태).
3. 서버 상태로 표현할 수 없다.

- 액세스 토큰 등 민감 정보는 **메모리에만 보관**한다. 영구 저장소(localStorage 등)에 저장하면 XSS 시 즉시 탈취된다.

---

## 3. URL 상태

- 목록 페이지의 필터·페이지·정렬은 URL 쿼리스트링에 저장한다. 이렇게 하면 새로고침·공유·뒤로가기가 자연스럽게 동작한다.
- 실제 적용값(API 요청에 사용)과 임시 입력값(아직 URL 미반영)을 분리해, "검색" 실행 전까지는 쿼리가 바뀌지 않도록 한다.

---

## 4. 폼 상태

- 모든 폼 입력은 폼 상태 라이브러리 + 스키마 검증으로 관리한다. 폼 필드를 지역 상태로 하나씩 관리하지 않는다.
- 폼 검증 스키마는 별도 스키마 레이어에 선언한다.
- 폼 입력 오류는 필드 아래 인라인으로 표시한다.

---

## 5. 지역 상태

- 컴포넌트 내부에서만 사용되는 상태(모달 open/close, 선택된 행, 탭 인덱스 등)는 지역 상태로 관리한다. 단일 컴포넌트 한정이면 전역 스토어나 Context 로 끌어올리지 않는다.
