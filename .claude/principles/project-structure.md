# 프로젝트 구조 (Iron Rule)

- **기존 모듈/디렉토리 경계를 존중한다.** 저장소의 구조는 이미 안정화되어 있다. 도메인 간 경계를 넘어 코드를 뒤섞지 않는다.
- 새 파일은 **가장 가까운 기능 도메인 근처**에 배치한다. 기존 도메인 폴더가 있으면 새 폴더/패키지를 만들지 않는다.
- **강력한 근거 없이 최상위 디렉토리/패키지를 새로 만들지 않는다.** 도메인 목록은 완만하게만 확장한다. 추가가 꼭 필요하면 PR 에 사유를 남긴다.
- **아키텍처 맵과 일관된 의존성 방향을 유지한다.**
  - 경계 → 서비스 → 인프라/영속 의 단방향 의존을 따른다. 역방향 호출은 이벤트 또는 명시적 우회로 처리한다.
  - 순수 타입/모델 레이어는 다른 레이어를 참조하지 않는다.
  - 같은 레이어 간 가로지르기는 제한적으로 허용하되, **역방향 참조**는 금지한다.
- **경계 스키마 원칙:** 외부에 노출하는 요청/응답 스키마는 경계 레이어에서만 노출하고, 내부 도메인 객체는 내부에서만 유통한다. 경계를 넘을 때는 명시적으로 변환한다.
- **구조가 불명확한 경우** 다음 순서로 1차 참조한다:
  1. 아키텍처·레이어 지도 문서
  2. 명명·레이어·API 규약 문서
  3. 스택·설정 기준 문서
  4. 기존 도메인의 실제 코드

---

## 새 모듈 배치 원칙

1. **기능 도메인을 먼저 파악한다.** 어느 레이어의 책임인지 판단한 뒤 배치한다.
2. **기존 패키지/폴더 안에 파일을 추가**한다. 새 서브패키지·최상위 패키지 신설은 강력한 근거가 없으면 금지한다.
3. 도메인이 커져 여러 하위 개념으로 나뉘는 경우, **하위 슬라이스마다 동일한 서브구조를 반복**한다. 최상위에 layer 폴더를 두고 그 아래 도메인을 섞는 역구조는 금지한다.
