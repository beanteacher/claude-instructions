# 코드 스타일 (Iron Rule)

- **프로젝트 포매터/린터 설정을 최우선으로 따른다.** 그 결과가 항상 우선이며, 설정은 한 곳에 명시하고 CI 에서 강제한다.
- **영리한 추상화보다 명확한 이름의 작은 함수를 선호한다.**
  - 제네릭 3중 nesting 으로 만든 한 줄 표현보다, 이름이 설명해 주는 private helper 2~3개로 나누는 쪽이 좋다.
  - 검증 단계를 의미 단위 헬퍼(`validate...` / `check...` / `resolve...`)로 분리하는 스타일을 따른다.
  - **함수 하나 = 한 가지 책임.** 여러 관심사를 합치지 않는다.
- **브라운필드 파일에서는 diff 표면을 최소화한다.**
  - 기존 파일의 포매팅을 대량으로 바꾸지 않는다. 같은 PR 에 로직 변경과 포매팅이 섞이면 리뷰 반려.
  - 포매팅만 정리하고 싶다면 별도 `style: ...` 커밋으로 분리한다.
- **승인된 패턴 우선 재사용.**
  - 규칙·예시 문서에 기재된 패턴 외의 구조를 도입하려면 PR 본문에 근거를 남긴다.
- **주석은 WHY 를 남긴다.** WHAT 은 잘 지어진 이름과 타입이 대신한다. 불필요한 `TODO` / `FIXME` 를 남기려면 이슈 링크·복구 계획을 같이 적는다.
- **타입을 숨기지 않는다.** 중간 변수·반환 타입에서 타입을 모호하게 두지 않고 명시한다.
- **와일드카드 import 금지.** 명시적 named import 를 사용한다.

---

## 안티패턴 금지 목록 (공통)

- 모호한 타입(`any` 등)으로 타입 검증을 우회하기 → 구체 타입 명시 또는 타입 좁히기.
- 예외를 표준 예외 계층 없이 그대로 던지기 → 도메인 예외 + 공통 에러 코드 사용.
- 공용 빈/인스턴스(HTTP 클라이언트, 직렬화기 등)를 컴포넌트·함수 안에서 중복 생성하기 → 공용 인스턴스 재사용.
- 에러를 조용히 삼키기 (`catch (e) {}`) → 의도적 무시 시 WHY 주석 + 경고 로깅.
- 와일드카드 import 남용.
- 브라운필드 파일에서 로직 변경과 대규모 포매팅 정리를 한 커밋에 섞기.
