# 만들기 전에 검색하기 (Iron Rule)

- 익숙하지 않은 인프라/프레임워크/UI 패턴을 도입하기 전에 **기존 솔루션을 먼저 확인**한다. 저장소에 이미 갖춰진 공통 자산을 중복 구현하지 않는다. 전형적 예:
  - **HTTP 클라이언트** → 공용 인스턴스/설정
  - **인증·세션 식별** → 공용 인증 유틸/가드
  - **예외 표준** → 공통 예외 + 에러 코드 + 핸들러
  - **페이지/목록 응답** → 공통 응답 래퍼 타입
  - **공통 영속/조회 기능** → 공용 베이스 / 헬퍼
  - **공용 UI·유틸 컴포넌트** → 기존 공유 컴포넌트/유틸
- 제약 조건을 충족하는 경우 **검증된 로컬 패턴을 재사용**한다. 규칙·패턴 예시 문서가 1차 참조 대상이다.
- 비표준 접근 방식(새 HTTP 라이브러리, 대체 ORM/상태 관리, 커스텀 트랜잭션 매니저/라우터 등)을 선택하는 경우 **원칙 기반 근거를 문서화**하고 PR 본문에 기록한다.
- 다음을 구분해서 판단한다:
  - **검증된 패턴** — 이미 저장소나 생태계에서 일반화된 구성. → 그대로 사용.
  - **검토가 필요한 새로운/인기 있는 패턴** — 도입 시 팀 리뷰 필요.
  - **특정 제약 조건으로 정당화되는 독자적 접근 방식** — 프로젝트 고유 요건으로 만들어진 구조.
- 외부 라이브러리 추가 판단 흐름:
  1. 사용 중인 프레임워크/생태계에 이미 있는가? → 있으면 그걸 쓴다.
  2. 기존 의존성으로 짧게(수십 줄 안에) 해결되는가? → 그러면 라이브러리를 추가하지 않는다.
  3. 추가해야 한다면 기존 의존성과 충돌하지 않는지, (프론트의 경우) 번들 사이즈 영향이 허용 범위인지 먼저 확인한다.

## 새 구성요소 만들기 전 체크리스트

```
(1) 공유 UI / 컴포넌트 디렉토리 검색: 동일 역할의 컴포넌트가 이미 있는가?
(2) 훅 / 서비스 디렉토리 검색: 동일 로직이 이미 있는가?
(3) 유틸/공통 디렉토리 검색: 같은 유틸리티 함수가 이미 있는가?
(4) 비슷한 도메인 구현 살펴보기: 같은 패턴으로 구현된 사례가 있는가?
```
