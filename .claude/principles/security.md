# 보안 (Iron Rule)

- **시크릿이나 자격증명을 절대 하드코딩하지 않는다.** JWT secret, DB 비밀번호, 외부 API 키 등은 환경변수/설정 주입으로만 공급하고, 운영 값은 배포 파이프라인의 환경변수/Secrets 로 공급한다. 코드에 문자열 리터럴로 하드코딩하지 않는다.
- **공개 경로는 명시된 것만 공개다.** 새 엔드포인트를 공개하려면 공개 경로 목록을 직접 수정하고, PR 본문에 공개 사유를 남긴다. 기본은 항상 인증 요구.
- **사용자 제어 데이터에 입력 검증과 출력 인코딩을 적용한다.** 원시 문자열을 쿼리에 직접 concat 하지 않는다(바인딩 파라미터/ORM 사용). 응답에 사용자 입력을 그대로 echo 할 때는 escape 처리한다.
- **인증(auth)과 인가(authorization)를 암묵적 동작이 아닌 명시적 인수 기준으로 처리한다.** 보호 라우트의 단일 진입점(가드/의존성)을 통과시키고, 개별 컴포넌트/페이지에서 인증 여부를 중복 체크하지 않는다.
- **인증/인가 경로를 수정할 때는 해당 변경에 대한 테스트를 반드시 추가한다.** 공개 경로 추가, 필터/가드 순서 변경, 토큰 TTL 조정은 보안 영향 범위가 크므로 독립 커밋으로 분리한다.
- **민감 정보 로깅 금지.** 토큰·비밀번호·OTP·검증 코드는 마스킹을 거쳐야 하고, 디버그 레벨이라도 평문으로 남기지 않는다.
- **암호화/복호화는 공통 유틸을 경유한다.** 서비스 내부에서 저수준 암호화 API 를 직접 다루지 않는다.
- **최소 권한 원칙을 적용한다.** DB 계정·외부 API 키에 필요한 권한만 부여하고, 읽기/쓰기 분리가 가능하면 분리한다.
- **공개 경로(webhook 등)는 진입 직후 서명/토큰 검증을 수행한다.** 공격자가 임의 payload 를 주입할 수 있다는 전제로 작성한다.

## OWASP Top 10 PR 체크리스트

PR 에 보안 관련 변경이 있을 때 아래 카테고리를 확인한다.

| 카테고리 | 확인 항목 |
|---------|---------|
| Injection (SQL 등) | 바인딩 파라미터/ORM 사용, raw 문자열 concat 없음 |
| Broken Access Control | 보호 경로가 가드/인증을 통과, 공개 사유 명시 없는 미인증 경로 없음 |
| Sensitive Data Exposure | 시크릿 하드코딩 없음, 로그에 토큰·패스워드 평문 없음 |
| Security Misconfiguration | 디버그 플래그 운영에서 비활성, 기본 자격증명 변경 |
