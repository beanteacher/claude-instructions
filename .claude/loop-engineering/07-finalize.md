# 7. FINALIZE — 통합 검증과 보고

> 모든 작업이 끝난 뒤 **전체를 통합 검증하고, 증거와 함께 보고** 하는 종료 단계.
> 핵심 원칙: **증거 기반 보고.** 실패가 있으면 숨기지 않고 출력과 함께 보고한다.

[← 06-checkpoint](./06-checkpoint.md) · [개요](./README.md)

---

## 이 단계의 목표

| 입력 | 행동 | 산출물 | 가드 |
|------|------|--------|------|
| 모든 작업 done | 전체 빌드/테스트 + 완료 체크리스트 | 증거 기반 완료 보고 | 미충족 DoD 있으면 "완료" 선언 금지 |

---

## HOW 1 — 통합 검증 (개별 GREEN ≠ 전체 GREEN)

각 작업이 단위로 통과했어도, 합쳤을 때 깨질 수 있다. 전체를 한 번 돌린다.

```
① 전체 컴파일        ./gradlew compileJava
② 전체 테스트        ./gradlew test
③ 빌드 산출물 생성    ./gradlew bootJar   (실제 패키징까지 되는지)
④ (선택) 통합/E2E    @SpringBootTest, 컨테이너 기동 등
```

> 단위 루프에서는 비싸서 미뤘던 "전체" 를 여기서 정산한다. 통합 단계에서 처음 드러나는 것: 빈 충돌, 순환 의존, 설정 누락, 직렬화 문제.

---

## HOW 2 — DoD 정산 (역추적)

[00-intake](./00-intake.md) 의 DoD 한 줄 한 줄을 **충족 증거와 매핑** 한다.

```yaml
dod_audit:
  - dod: "PENDING 취소 시 CANCELED"
    proven_by: "OrderServiceTest.취소_PENDING이면_CANCELED  ✅"
  - dod: "PAID 취소 시 409 + ORDER_NOT_CANCELABLE"
    proven_by: "OrderControllerTest.취소_PAID면_409  ✅"
  - dod: "타인 주문 403"
    proven_by: "OrderControllerTest.취소_타인주문이면_403  ✅"
```

DoD 중 하나라도 `proven_by` 가 비면 **완료가 아니다.** PLAN 으로 돌아간다.

---

## HOW 3 — 완료 체크리스트 (범용 + 부수효과)

기능 외에 시스템 정합성을 점검한다.

- [ ] **전체 테스트 GREEN** (단위 통과 ≠ 전체 통과)
- [ ] **빌드 산출물 생성 성공**
- [ ] **DB 스키마 변경 시 DDL 소스 동기화** (운영/참조 양쪽)
- [ ] **공개 경로/권한 변경이 의도대로** 반영
- [ ] **부수효과**(파일 생성, 외부 동기화, 이벤트 발행)가 동작/누락 없음
- [ ] **하위 호환** 깨짐 없음 (또는 마이그레이션 동반)
- [ ] **임시 코드·디버그 로그·죽은 코드** 제거
- [ ] **문서/주석** 이 변경과 일치 (오해 유발 잔존 없음)

---

## HOW 4 — 증거 기반 보고 형식

보고는 "했다" 가 아니라 **"이렇게 확인됐다"** 로 쓴다.

```markdown
## 완료: 주문 취소 API

### 변경 요약
- OrderService 취소 상태전이 + 가드(PAID/타인) 추가
- OrderController DELETE /orders/{id} 추가

### 검증 (증거)
- ./gradlew test  →  142 passed, 0 failed
- ./gradlew bootJar  →  BUILD SUCCESSFUL (oam.jar 생성)
- DoD 3/3 충족 (테스트로 입증, 위 dod_audit 참조)

### 주의/후속
- 환불·재고 복원은 out_of_scope (별도 작업 필요)
- 동시 취소 경합은 미처리 — 후속 카드로 분리 권장
```

---

## HOW 5 — 정직한 종료의 3가지 형태

```
① 완전 완료    모든 DoD 충족 + 전체 GREEN
                 → 증거와 함께 완료 보고

② 부분 완료    일부 DoD 충족, 나머지 막힘/범위 밖
                 → 된 것/안 된 것 명확히 구분해 보고 (안 된 걸 됐다고 하지 않음)

③ 차단(blocked) 에스컬레이션으로 멈춤 (05-verify HOW 4)
                 → 막힌 지점 + 시도 가설 + 마지막 출력 보고
```

세 경우 모두 **테스트 실패가 남아 있으면 그 출력을 그대로 첨부** 한다. 숨기지 않는다.

## 안티패턴

| 안티패턴 | 결과 | 대신 |
|----------|------|------|
| 단위만 보고 "완료" | 통합에서 터짐 | 전체 빌드/테스트 정산 |
| DoD 정산 생략 | 빠진 요구사항 미인지 | DoD↔증거 역추적 |
| 실패 숨기고 보고 | 신뢰 붕괴 | 실패도 출력과 함께 |
| "거의 다 됐다" 모호 보고 | 후속자가 상태 모름 | 된 것/안 된 것 명확 구분 |
| 스키마/부수효과 점검 누락 | 운영에서 사고 | 완료 체크리스트 |
