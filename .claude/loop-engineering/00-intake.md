# 0. INTAKE — 요구사항 명료화

> 사용자의 자연어 요청을 **측정 가능한 완료의 정의(Definition of Done)** 로 변환하는 단계.
> 여기서 DoD 가 흐리면 이후 모든 단계가 흔들린다. 루프의 입구이자 가장 값싼 교정 지점.

[← 개요](./README.md) · [다음: 01-plan →](./01-plan.md)

---

## 이 단계의 목표

| 입력 | 행동 | 산출물 | 가드(중단 조건) |
|------|------|--------|------------------|
| 자연어 요청 | 모호성 해소 + DoD 확정 | 측정 가능한 수락 기준 목록 | DoD 가 측정 불가능하면 루프 시작 금지 |

---

## HOW 1 — 요청을 4가지로 분해한다

들어온 요청을 다음 슬롯으로 채워본다. 빈 슬롯이 곧 질문거리다.

```
WHAT   무엇을 만드는가         (대상 리소스/엔드포인트/기능)
WHY    왜 필요한가            (해결하려는 문제 — 스코프 판단 기준)
DONE   무엇이 끝의 신호인가    (관찰 가능한 결과 = DoD)
BOUND  무엇은 하지 않는가      (명시적 비범위 — 스코프 크리프 방지)
```

예시:

```
요청: "주문 취소 기능 만들어줘"

WHAT  : DELETE /orders/{id} 또는 상태 전이 API
WHY   : 고객이 결제 전 주문을 되돌릴 수 있어야 함
DONE  : ??? ← 비어있음 → 질문 필요
BOUND : ??? ← 환불 연동까지인가? 재고 복원 포함인가? → 질문 필요
```

---

## HOW 2 — DoD 는 "관찰 가능한 문장" 으로 쓴다

DoD 한 줄은 **나중에 OBSERVE 단계에서 테스트로 그대로 옮길 수 있어야** 한다.

| ❌ 측정 불가 | ✅ 측정 가능 (테스트로 직결) |
|--------------|------------------------------|
| 주문 취소를 잘 처리한다 | `PENDING` 상태 주문에 취소 요청 시 상태가 `CANCELED` 로 바뀐다 |
| 에러를 적절히 던진다 | 이미 `PAID` 인 주문 취소 시 `409 + ErrorCode.ORDER_NOT_CANCELABLE` |
| 빠르게 응답한다 | 단건 취소 p95 < 200ms (해당 시) |
| 권한을 체크한다 | 본인 주문이 아니면 `403` |

**DoD 문장 템플릿**:
```
[전제 상태] 에서 [행동] 하면 [관찰 가능한 결과] 가 된다.
```

---

## HOW 3 — 명료화 질문은 "값싼 것만, 묶어서"

질문은 비용이다. 다음 기준으로 거른다.

```
질문해야 하는가?
├─ 답에 따라 설계가 갈리는가?        → YES면 질문
│    (예: "소프트 삭제 vs 하드 삭제?")
├─ 합리적 기본값이 있는가?            → YES면 질문 말고 기본값 채택 + 명시
│    (예: 정렬 순서 미지정 → 최신순 기본)
└─ 코드/문서에서 확인 가능한가?       → YES면 질문 말고 탐색(02-context)
     (예: "기존 응답 포맷이 뭐죠?" → 직접 찾는다)
```

- 질문이 여러 개면 **한 번에 묶어서** 묻는다 (왕복 비용 절감).
- 기본값으로 진행할 땐 **"X로 가정하고 진행합니다"** 라고 선언한다 (나중에 뒤집기 쉽게).

---

## HOW 4 — 모호성 신호(red flag) 체크리스트

요청에 아래가 보이면 DoD 가 덜 익은 것이다.

- [ ] "잘", "적절히", "알아서", "최적화" 같은 **부사**만 있고 기준 수치가 없다
- [ ] 동사는 있는데 **대상 리소스/엔드포인트가 불명확**하다
- [ ] **성공과 실패를 가르는 경계 조건**이 안 적혀 있다 (빈 값, 중복, 권한, 동시성)
- [ ] **비범위(하지 않을 것)** 가 없어 무한 확장 가능하다
- [ ] 여러 기능이 한 문장에 뭉쳐 있다 (→ PLAN 에서 쪼개야 한다는 신호)

## 산출물 예시 (다음 단계로 넘기는 형태)

```yaml
goal: "주문 취소 API"
context_note: "결제 전 되돌리기 — 환불/재고는 별도 작업"
definition_of_done:
  - "PENDING 주문 취소 시 상태가 CANCELED 로 전이된다"
  - "PAID 주문 취소 시 409 + ErrorCode.ORDER_NOT_CANCELABLE"
  - "존재하지 않는 주문 취소 시 404"
  - "본인 주문이 아니면 403"
out_of_scope:
  - "환불 PG 연동"
  - "재고 복원"
assumptions:
  - "취소는 소프트 상태 전이 (레코드 삭제 안 함)"
```

이 산출물이 [01-plan](./01-plan.md) 의 입력이 된다.

## 안티패턴

| 안티패턴 | 결과 | 대신 |
|----------|------|------|
| DoD 없이 바로 코딩 시작 | 무엇이 끝인지 몰라 루프가 안 멈춤 | 측정 가능한 DoD 먼저 |
| 모든 걸 다 질문 | 왕복 폭증, 사용자 피로 | 설계가 갈리는 것만, 묶어서 |
| 기본값 가정을 침묵 | 나중에 뒤집기 비쌈 | "X로 가정" 명시 선언 |
| 비범위 미정의 | 스코프 무한 확장 | out_of_scope 명시 |
