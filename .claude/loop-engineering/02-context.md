## HOW 1 — 3종 탐색을 나눠서 한다

탐색은 목적이 다른 3가지를 구분해서 수행한다.

```
① 재사용 탐색   "이 기능, 이미 있나?"
     → 공용 빈, 베이스 클래스, 유틸, 비슷한 도메인 구현

② 규칙 탐색     "이 코드, 어떤 컨벤션을 따라야 하나?"
     → 네이밍, 레이어 책임, 예외 처리, 매핑 방식, 커밋 규칙

③ 영향 탐색     "이걸 바꾸면 어디가 깨지나?"
     → 호출처, 의존 모듈, 스키마, 공개 경로, 부수효과
```

---

## HOW 2 — 검색 전략 (넓게 → 좁게)

| 찾는 것 | 검색 방식 | 예 |
|---------|-----------|-----|
| 심볼/클래스 위치 | 이름·패턴 검색 | `OrderService`, `*RepositoryImpl` |
| "비슷한 게 있나" | 기능 키워드 검색 | "취소", "cancel", "상태전이" |
| 호출처 (영향 범위) | 참조 찾기 (find references) | 메서드 호출 지점 전부 |
| 컨벤션 근거 | 규칙 문서 + 기존 동일 레이어 코드 | 가장 최근/대표 도메인을 레퍼런스로 |

> 팁: **가장 잘 정리된 동일 계층 코드 하나를 "레퍼런스 구현"** 으로 삼아 그 모양을 복제한다. 새 패턴을 발명하지 않는다.

---

## HOW 3 — 재사용 판단 흐름

```
새 컴포넌트를 만들고 싶다.
├─ 똑같은 것이 이미 있나?            → YES → 그것을 쓴다 (만들지 않음)
├─ 거의 같은데 약간 다른가?          → 기존을 확장/파라미터화 가능한가?
│      ├─ YES → 기존을 일반화         (분기 메서드 양산 금지)
│      └─ NO  → 신규 (단, 명명·구조는 기존 관행 복제)
└─ 완전히 새로운 개념인가?           → 신규, 단 레이어 규칙 준수
```

특히 **thread-safe 재사용 컴포넌트**(HTTP 클라이언트, 직렬화기, 템플릿 엔진, 암호화기)는 서비스 안에서 `new` 하지 않고 **이미 등록된 공용 빈/싱글톤을 주입**받는다.

---

## HOW 4 — 영향 범위(blast radius) 체크리스트

작업 전, 아래가 이 변경에 걸리는지 확인한다.

- [ ] 이 메서드/필드의 **호출처가 몇 곳인가** (시그니처 변경 시 전부 영향)
- [ ] **DB 스키마**가 바뀌는가 → DDL 소스 동기화 필요
- [ ] **공개 경로/권한**이 바뀌는가 → 보안 설정 + 별도 커밋
- [ ] **부수효과**(파일 생성, 외부 동기화, 이벤트)를 건드리는가 → 트랜잭션 경계 확인
- [ ] **하위 호환**을 깨는가 (API 응답 형태, enum 값) → 마이그레이션 고려
- [ ] 동일 변경이 **여러 곳에 흩어져** 있는가 → 한 곳으로 모을 수 있나

## 산출물 — "컨텍스트 묶음"

ACT 단계로 넘기는 한 장 요약:

```yaml
task: "OrderService 취소 상태전이"

reuse:
  - "BaseRepository.findByIdOrThrow 사용 (없음 처리 중복 작성 금지)"
  - "기존 OrderManageService 가 레퍼런스 구조"
  - "예외는 WiseException + ErrorCode (커스텀 예외 클래스 만들지 않음)"

rules:
  - "쓰기 메서드 @Transactional(rollbackFor=Exception.class)"
  - "Entity→DTO 변환은 MapStruct 매퍼 사용"

blast_radius:
  - "OrderStatus enum 에 CANCELED 추가 → 컨버터/스키마 영향 확인"
  - "취소 후 알림 발행 필요 → 이벤트 발행(afterCommit)로 분리"

reference_files:
  - "기존: order/manage/service/OrderManageService.java"
  - "규칙: conventions.md, mapping.md"
```

## 안티패턴

| 안티패턴 | 결과 | 대신 |
|----------|------|------|
| 탐색 없이 바로 작성 | 중복 구현, 컨벤션 위반 | 3종 탐색 선행 |
| 공용 빈 무시하고 `new` | 설정 분산, 메모리 낭비 | 등록된 싱글톤 주입 |
| 새 패턴 발명 | 일관성 붕괴 | 레퍼런스 구현 복제 |
| 영향 범위 미확인 | 다른 곳이 조용히 깨짐 | blast radius 체크리스트 |
| 분기마다 메서드 양산 | 유지보수 폭증 | enum 파라미터로 일반화 |
