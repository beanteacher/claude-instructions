# 1. PLAN — 작업 분해와 정렬

> DoD 를 **한 사이클에 하나씩 처리 가능한 작업 단위** 로 쪼개고, 의존성 순서로 정렬하는 단계.
> 핵심 질문: "이걸 어떻게 가장 작은 검증 가능 조각들로 나눌까?"

## HOW 1 — 수직 슬라이스로 쪼갠다 (수평 금지)

기능은 **얇고 끝까지 관통하는 수직 슬라이스** 로 나눈다. "레이어 전체를 한 번에" 가 아니라 "한 시나리오를 끝까지".

```
❌ 수평 분해 (한 번에 한 레이어 전체)
   1. 모든 엔티티 작성
   2. 모든 리포지토리 작성
   3. 모든 서비스 작성
   4. 모든 컨트롤러 작성
   → 3번 끝날 때까지 아무것도 동작하지 않음, 통합 리스크가 끝에 몰림

✅ 수직 슬라이스 (한 시나리오를 끝까지)
   슬라이스 A: "정상 취소" 경로만 Repository→Service→Controller 관통
   슬라이스 B: "이미 결제됨 → 409" 경로 추가
   슬라이스 C: "권한 없음 → 403" 경로 추가
   → 각 슬라이스 끝에 동작하는 것이 생긴다
```

---

## HOW 2 — 백엔드 TDD 계층 순서 (슬라이스 내부)

한 수직 슬라이스 안에서는 **안쪽(데이터)부터 바깥쪽(API)으로** 테스트 우선 쌓기.

```
1. RepositoryTest   (슬라이스/DB 슬라이스 테스트)   ─ RED
2. Repository (+Entity)                            ─ 최소 구현 → GREEN
3. ServiceTest      (단위 테스트, mock)            ─ RED
4. Service                                         ─ 최소 구현 → GREEN
5. ControllerTest   (웹 슬라이스 테스트)           ─ RED
6. Controller (+DTO)                               ─ 최소 구현 → GREEN
7. (선택) IntegrationTest                          ─ 계층 통합 검증
```

> 왜 안쪽부터? 바깥(Controller)부터 만들면 아래가 mock 투성이가 되고, 정작 핵심 도메인 로직 검증이 뒤로 밀린다. 데이터·도메인을 먼저 굳히면 위 레이어는 그 위에 얇게 얹힌다.

## HOW 3 — 작업 크기 기준 ("한 사이클에 들어가는가")

한 작업 카드는 아래를 만족해야 한다. 안 되면 더 쪼갠다.

- [ ] **하나의 RED→GREEN 사이클**로 닫힌다 (테스트 1~소수 개)
- [ ] **하나의 명확한 검증 명령**이 있다 (`test --tests X`)
- [ ] 변경 파일이 대략 **1~3개** 수준 (그 이상이면 책임이 섞였을 가능성)
- [ ] 한 문장으로 제목이 써진다 ("OrderService 취소 상태전이")
- [ ] 다른 작업과 **독립적으로 커밋** 가능하다

---

## HOW 4 — 의존성 정렬과 병렬 식별

```
1) 각 작업의 선행 의존을 적는다
   - "ServiceTest" 는 "Entity/Repository 시그니처" 에 의존
   - "ControllerTest" 는 "Service 시그니처" 에 의존

2) 위상 정렬 → 실행 순서 확정

3) 서로 의존 없는 작업은 병렬 가능 표시
   - 예: "주문 도메인" 과 "독립적인 알림 템플릿" → 병렬
   - 단, 같은 파일을 건드리는 작업은 병렬 금지 (충돌)
```

---

## 작업 카드 템플릿

```yaml
- id: 3
  title: "OrderService 취소 상태전이"
  layer: service
  depends_on: [2]                # Repository 시그니처 필요
  dod_ref:                       # 어떤 DoD 를 만족시키는가
    - "PENDING 주문 취소 시 CANCELED"
    - "PAID 주문 취소 시 409"
  verify_cmd: "./gradlew test --tests *.OrderServiceTest"
  risk: "동시 취소 요청 경합 — 후속 작업으로 분리"
  status: pending
```

`risk` 칸은 발견 즉시 적되, 당장 처리할지/뒤로 뺄지는 명시한다.

---

## HOW 5 — 리스크·미지수 플래그

PLAN 단계에서 미리 깃발을 꽂아두면 ACT 에서 당황하지 않는다.

- **외부 의존** (PG, 메시지 큐, 외부 API) → mock 경계 어디에 둘지 미리 결정
- **스키마 변경** → DDL 동기화 작업을 별도 카드로
- **공개 경로/보안 변경** → 별도 커밋 + 리뷰 포인트 표시
- **부수효과** (파일 생성, peer 동기화, 이벤트 발행) → 트랜잭션 경계와 함께 표시

## 안티패턴

| 안티패턴 | 결과 | 대신 |
|----------|------|------|
| 수평 분해 (레이어 일괄) | 통합 리스크가 끝에 몰림 | 수직 슬라이스 |
| 거대 작업 1개 | 실패 격리 불가, 커밋 불가 | 한 사이클 크기로 쪼갬 |
| 의존성 무시한 순서 | 컴파일 안 되는 중간 상태 | 위상 정렬 |
| 계획에 검증 명령 누락 | OBSERVE 에서 뭘 돌릴지 모름 | 카드마다 `verify_cmd` |
| "나중에 바뀔까봐" 추측 분리 | 과설계 | DoD 에 있는 것만 계획 |
