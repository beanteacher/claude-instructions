# 3. ACT — 한 단위 실행

> 작업 카드 **하나만** 꺼내 실제 코드를 쓰는 단계.
> 핵심 원칙: **한 사이클 = 한 기능**, **테스트 먼저(TDD)**, **최소 구현**.

[← 02-context](./02-context.md) · [개요](./README.md) · [다음: 04-observe →](./04-observe.md)

---

## 이 단계의 목표

| 입력 | 행동 | 산출물 | 가드 |
|------|------|--------|------|
| 컨텍스트 묶음 + 작업 카드 1개 | RED→GREEN 1사이클 실행 | 변경된 코드 (테스트 + 구현) | 테스트 없이 프로덕션 코드 쓰지 않음 |

---

## HOW 1 — RED → GREEN → REFACTOR

```
① RED      실패하는 테스트를 먼저 쓴다 → 실행해서 "실패" 를 눈으로 확인
              (첫 실행에서 통과하면 테스트가 잘못된 것 — 04-observe 에서 검증)

② GREEN    그 테스트를 통과시키는 "최소" 구현만 쓴다
              (DoD 에 없는 기능을 미리 만들지 않는다)

③ REFACTOR 테스트가 GREEN 인 상태를 유지하며 중복 제거·정리
              (구조 개선은 동작 보존 — 여기서 컨벤션에 맞춘다)
```

각 화살표 사이에 [OBSERVE](./04-observe.md) 가 끼어든다. ACT 와 OBSERVE 는 사실상 한 호흡.

---

## HOW 2 — "최소 구현" 의 경계

GREEN 을 만드는 데 **꼭 필요한 코드만** 쓴다.

```
✅ 최소 구현
   - 테스트가 요구하는 분기/필드만 추가
   - 가짜 상수라도 테스트가 그것만 검증하면 일단 통과시키고 다음 테스트로 압박

❌ 과잉 구현 (YAGNI 위반)
   - "나중에 쓸까봐" 옵션 파라미터 추가
   - DoD 에 없는 케이스 미리 처리
   - 추측성 추상화 계층 (인터페이스 분리 등) 선제 도입
```

> 추상화(인터페이스+구현 분리 등)는 **실제로 두 번째 구현이 필요해질 때** 도입한다. 한 개뿐인데 미리 인터페이스를 파지 않는다.

---

## HOW 3 — diff 최소화 원칙

- 작업과 **무관한 포맷팅/리네이밍을 섞지 않는다** (리뷰·롤백이 어려워진다).
- 기존 코드 스타일(들여쓰기, 네이밍, 주석 밀도)을 **그대로 따라 쓴다**. 주변 코드처럼 보이게.
- 한 작업의 변경은 한 커밋으로 닫히게 유지 (→ [06-checkpoint](./06-checkpoint.md)).
- 무관한 개선거리를 발견하면 **메모만 남기고** 지금 건드리지 않는다.

---

## HOW 4 — 레이어별 ACT 체크 (백엔드 예)

| 레이어 | 이 단계에서 쓰는 것 | 하지 말 것 |
|--------|---------------------|------------|
| Entity | 상태 + 도메인 메서드, 팩토리 | 요청/응답 어노테이션 |
| Repository | 쿼리 메서드 | 비즈니스 로직 |
| Service | 비즈니스 로직 + 트랜잭션 경계 | DB 직접 접근, 컨트롤러 의존 |
| Mapper | Entity↔DTO 변환만 | 분기·계산·검증 |
| Controller | 요청 수신 + DTO 반환 | 비즈니스 로직, 엔티티 직접 반환 |

부수효과(파일 생성, 외부 동기화, 알림)는 서비스에서 직접 호출하지 않고 **이벤트 발행 → afterCommit 리스너** 로 분리 (해당 패턴을 쓰는 프로젝트의 경우).

---

## HOW 5 — TDD 위반을 인지했을 때 (복구 절차)

테스트 없이 프로덕션 코드를 먼저 써버린 걸 깨달으면:

```
1. 즉시 멈춘다.
2. 테스트 없이 작성된 코드를 식별한다.
3. (원칙) 그 코드를 지우고 RED 테스트부터 다시 시작.
4. (예외 — 삭제가 비현실적) 누락된 테스트를 즉시 보충하고,
   기존 코드가 통과 못 하면 코드를 고쳐 GREEN 으로 맞춘다.
   단, 다음 사이클부터는 원칙(테스트 우선)을 복원한다.
```

---

## 안티패턴

| 안티패턴 | 결과 | 대신 |
|----------|------|------|
| 여러 기능 동시 구현 | 실패 원인 격리 불가 | 한 사이클 = 한 기능 |
| 테스트 후행 ("일단 만들고") | 커버리지 맞추기 테스트로 전락 | RED 먼저 |
| 과잉 구현 (YAGNI) | 죽은 코드·복잡도 | DoD 가 요구하는 만큼만 |
| 무관한 리팩터 섞기 | 리뷰/롤백 곤란 | diff 최소화, 메모만 |
| 주변과 다른 스타일 | 일관성 붕괴 | 기존 코드처럼 작성 |
