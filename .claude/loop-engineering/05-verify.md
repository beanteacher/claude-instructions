# 5. VERIFY — 검증 게이트와 분기 ★

> 관찰 결과(OBSERVE)를 보고 **"통과 / 재시도 / 에스컬레이션"** 을 결정하는 단계.
> 루프 전체에서 **가장 중요한 게이트.** 무한 루프 방지와 거짓 완료 방지가 여기서 일어난다.

[← 04-observe](./04-observe.md) · [개요](./README.md) · [다음: 06-checkpoint →](./06-checkpoint.md)

---

## 이 단계의 목표

| 입력 | 행동 | 산출물 | 가드 |
|------|------|--------|------|
| 관찰 결과 + 작업 DoD | 다중 게이트 통과 판정 + 분기 결정 | `(ok, reason, next_action)` | 같은 작업 재시도 MAX_RETRY 초과 시 에스컬레이션 |

---

## HOW 1 — 분기 결정 흐름

```
              ┌─────────────────────────────────┐
관찰 결과 ───→ │ G1. 컴파일/타입  통과?           │ No → ACT (가장 값싼 수정)
              ├─────────────────────────────────┤
              │ G2. 단위 테스트  GREEN?          │ No → 원인 분석 → ACT (attempts+1)
              ├─────────────────────────────────┤
              │ G3. 회귀(인접) 테스트  GREEN?    │ No → 회귀 발생 → ACT (attempts+1)
              ├─────────────────────────────────┤
              │ G4. 컨벤션/품질  위반 없음?      │ No → REFACTOR → ACT
              ├─────────────────────────────────┤
              │ G5. 보안/경계   문제 없음?       │ No → 수정 (보안은 통과 불가 항목)
              ├─────────────────────────────────┤
              │ G6. DoD 항목    충족?            │ No → 남은 케이스 → PLAN 보강
              └─────────────────────────────────┘
                          ↓ All pass
                    CHECKPOINT 로 진행

  ※ 어느 게이트든 재시도 시 attempts += 1.
    attempts >= MAX_RETRY → ESCALATE (아래 HOW 4)
```

게이트는 **값싼 것 → 비싼 것** 순서. 컴파일 깨졌는데 컨벤션을 보지 않는다.

---

## HOW 2 — 게이트별 "통과 기준"

| 게이트 | 통과 기준 | 판정 방법 |
|--------|-----------|-----------|
| G1 컴파일 | 에러 0 | 빌드 명령 exit 0 |
| G2 단위 | 대상 테스트 전부 GREEN | `test --tests X` 결과 |
| G3 회귀 | 영향 범위 테스트 GREEN | 호출처 테스트 |
| G4 컨벤션 | 레이어 책임·네이밍·매핑·예외 규칙 준수 | 규칙 대조 (린트+수동) |
| G5 보안 | 시크릿 노출·권한 누락·입력 검증 없음 0 | 체크리스트 (HOW 3) |
| G6 DoD | 해당 작업의 DoD 문장이 테스트로 입증됨 | DoD↔테스트 매핑 확인 |

> **G6 가 핵심**: "테스트가 통과했다" 와 "DoD 를 만족한다" 는 다르다. 통과한 테스트가 정작 DoD 문장을 검증하는지 대조한다.

---

## HOW 3 — 보안/경계 미니 체크리스트 (G5)

작업이 입력·권한·외부연동을 건드리면:

- [ ] 시크릿/키/접속정보가 코드·로그에 하드코딩되지 않았다
- [ ] 공개 경로 추가 시 의도적이며 권한 모델과 일치한다
- [ ] 외부 입력(요청 body, webhook)에 검증이 있다
- [ ] 권한 체크가 우회되지 않는다 (본인 리소스 확인 등)
- [ ] 에러 응답이 내부 구조(스택트레이스, 쿼리)를 노출하지 않는다

보안 게이트는 "나중에" 가 없다. 실패 시 통과 불가.

---

## HOW 4 — 재시도 vs 에스컬레이션 (무한 루프 방지)

```
실패 →  attempts < MAX_RETRY ?
        ├─ YES → 자기수정 후 재시도
        │         단, "다른 가설" 로 시도한다. 같은 수정 반복 금지.
        └─ NO  → ESCALATE
                 ├─ 진행을 멈춘다
                 ├─ 막힌 지점 + 시도한 가설들 + 마지막 출력을 보고
                 └─ 사람(또는 상위 에이전트)에게 판단을 넘긴다
```

**재시도가 의미 있으려면 가설이 매번 달라야 한다.** 같은 변경을 3번 반복하면 그건 1번 실패와 같다. 시도 로그를 남긴다:

```yaml
attempts:
  - try: 1
    hypothesis: "PAID 가드 분기 누락 → 추가"
    result: "여전히 실패 — 상태 비교가 == 가 아니라 equals 문제"
  - try: 2
    hypothesis: "enum 비교를 equals 로 교정"
    result: GREEN
```

**에스컬레이션은 실패가 아니다.** 무한히 헛도는 것보다 막힌 걸 정직하게 올리는 게 옳은 종료다.

---

## HOW 5 — 거짓 완료(false done) 방지

다음을 절대 "통과" 로 인정하지 않는다.

- 테스트를 비활성화/스킵해서 GREEN 으로 만든 것
- assert 를 약하게 고쳐(`assertNotNull` 만) 통과시킨 것
- 실패하는 테스트를 지워서 통과시킨 것
- DoD 케이스를 구현 안 하고 테스트만 통과시킨 것 (G6 미충족)

이런 게 보이면 게이트 실패로 되돌린다.

---

## 산출물

```yaml
verdict: pass            # pass | retry | escalate
gates:
  compile: pass
  unit: pass
  regression: pass
  convention: pass
  security: pass
  dod: pass
attempts_used: 2
note: "enum 비교 equals 교정으로 GREEN"
```

`pass` 면 [06-checkpoint](./06-checkpoint.md), `retry` 면 [03-act](./03-act.md), `escalate` 면 보고 후 종료.

## 안티패턴

| 안티패턴 | 결과 | 대신 |
|----------|------|------|
| 테스트만 통과 = 완료 | DoD 미충족인데 닫힘 | G6 에서 DoD↔테스트 대조 |
| 같은 수정 무한 반복 | 영원히 안 끝남 | MAX_RETRY + 가설 변경 |
| 에스컬레이션 회피 | 비용 폭증 | 막히면 정직하게 올린다 |
| 테스트 스킵/약화로 GREEN | 거짓 완료 | false done 으로 간주, 되돌림 |
| 보안 게이트 "나중에" | 취약점 유입 | G5 는 통과 불가 항목 |
