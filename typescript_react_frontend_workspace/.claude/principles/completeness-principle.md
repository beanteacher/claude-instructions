# 완전성 원칙 (Iron Rule)

- **증분 비용이 낮을 때는 완전하고 검증 가능한 구현을 선호한다.**
  - 한 도메인 기능을 추가하면서 `api/*-api.ts` / `model/dto/` / `hooks/react-query/` / `page/<domain>/` 까지 함께 쌓는다.
  - 라우트를 추가하면서 `app/**/page.tsx` 등록 + 사이드바 메뉴 항목 + 인증 가드 공개/보호 분류까지 같이 처리한다.
  - 보안 경로를 건드리면 `PUBLIC_ROUTES` + axios 인터셉터 로직 + 토스트 메시지까지 같이 업데이트한다.
- **엣지 케이스와 테스트가 간단한 경우 "80–90% 완료"를 미완성으로 취급한다.**
  - "일단 happy path 만 머지하고 edge case 는 나중에" 스타일 지양.
  - `any` 타입 잔류, 401 후 재로그인 흐름, 빈 배열 응답, 네트워크 오류 같은 **이 프로젝트의 전형적 엣지**는 작성 시점에 함께 고려한다.
- 다음을 구분한다:
  - **호수(lake)** — 지금 완료할 수 있는 한정된 작업. 예: "문서 목록 테이블에 컬렉션명 컬럼 추가". → 한 번에 끝까지 구현한다.
  - **바다(ocean)** — 명시적 범위 결정이 필요한 무한한 재작성/마이그레이션. 예: "전체 페이지 i18n 적용", "Server Component 로 전환". → 범위를 쪼개 단계화된 체크리스트로 정리하고, 매 단계마다 독립적으로 머지 가능한 상태로 끝낸다.
- **리뷰 시 회피 가능한 편법과 누락된 검증을 지적한다.**
  - `TODO` / `FIXME` 를 남기면서 타입·UI·테스트 중 하나를 건너뛴 코드는 리뷰에서 반려.
  - `as any` / `// @ts-ignore` 로 타입 에러를 덮은 코드는 **복구 계획**을 commit 메시지/PR body 에 명시하지 않으면 반려.
  - "일단 console.log 만 남기고 진짜 에러 처리는 다음 PR" 같은 스텁은 피한다 — 만들 거면 끝까지, 아니면 분리된 TODO 이슈로 추적.

## 본 프로젝트에서 자주 누락되는 항목 (체크리스트)

새 기능을 완료로 선언하기 전에 아래 항목을 모두 확인한다:

```
[ ] 타입 누수: any / unknown 미처리 필드가 남아 있지 않은가?
    (model/dto/, model/client/, model/schema/ 를 grep 해 확인)

[ ] React Query 키 충돌 / invalidation 누락:
    새 쿼리 키가 기존 키와 겹치지 않는가?
    뮤테이션 성공 후 관련 쿼리를 invalidateQueries(['domain']) 로 무효화했는가?

[ ] 토스트 메시지 누락 (성공/실패 둘 다):
    API 성공 시 toast({ title: "...", description: "..." }) 가 있는가?
    API 실패 시 toast({ variant: "destructive", ... }) 가 있는가?

[ ] 다국어 키 누락:
    i18n 미도입 상태 — 한글 하드코딩 허용. 도입 시점 결정 후 일괄 마이그레이션.
    (현 시점: 이 항목은 확인 불필요, 도입 PR 작성 시 별도 체크리스트 추가 예정)

[ ] 로딩 상태 UI:
    공용 테이블 컴포넌트 사용 시 isLoading prop 이 연결되어 있는가?
    Sheet/Form 에서 데이터 로딩 중 로딩 상태를 노출하는가?

[ ] 에러 상태 UI:
    공용 테이블 컴포넌트의 error prop 이 연결되어 있는가?
    axios 인터셉터의 toast 외에 페이지 레벨 fallback 이 필요한가?

[ ] 빈 상태 UI:
    공용 테이블 컴포넌트의 data 가 빈 배열일 때 EmptyState 가 노출되는가?

[ ] 모바일 뷰 점검:
    hooks/use-mobile.ts (useIsMobile) 로 분기가 필요한 레이아웃인가?

[ ] 다크모드 점검:
    next-themes + components/theme-toggle.tsx 기반으로 색상 토큰이 올바른가?
    Tailwind 유틸리티 클래스가 dark: 변형을 의식한 디자인 토큰을 쓰는가?

[ ] 폼 검증:
    react-hook-form + zod 스키마(model/schema/)가 연결되어 있는가?
    유효성 오류 메시지(FormMessage)가 사용자에게 노출되는가?

[ ] 사이드바 항목 추가:
    새 라우트가 생기면 사이드바 컴포넌트의 items 배열에 메뉴를 추가했는가?

[ ] 라우트 가드:
    새 라우트가 보호 대상이면 PUBLIC_ROUTES 에서 제외했는가?
    공개 라우트라면 PUBLIC_ROUTES 에 추가했는가?
    인증 가드(auth-guard) 가 해당 경로를 올바르게 통과/차단하는가?

[ ] 접근성:
    키보드 네비게이션이 가능한가? (Tab 순서, Enter/Space 활성화)
    아이콘 전용 버튼에 aria-label 이 있는가?
    포커스 링(focus ring)이 시각적으로 표시되는가?

[ ] 권한 (역할별 메뉴 노출):
    인증 스토어의 roles 정보로 조건부 렌더링이 필요한가?
    (프로젝트 권한 모델은 common/app-type.ts 또는 해당 타입 파일 확인)
```
