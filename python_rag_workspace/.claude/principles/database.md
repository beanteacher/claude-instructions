# 데이터베이스 규칙 (Iron Rule)

---

## 1. 연결 관리

- **커넥션 풀**을 사용한다. 요청마다 새 연결을 생성하지 않는다.
- 커넥션은 context manager(`with`) 또는 전용 헬퍼를 통해 획득하고, 반드시 반환한다.
- Pool 설정(min/max 커넥션 수, 타임아웃)은 환경변수로 주입한다.

---

## 2. 추가적(Additive) 마이그레이션 원칙

**브라운필드 운영 DB에서 안전한 배포를 위해 추가적 변경만 허용한다.**

- **컬럼 추가**: `NULL` 허용 + `DEFAULT` 값 전략. NOT NULL 컬럼을 기존 테이블에 즉시 추가하지 않는다.
- **컬럼 삭제/리네임**: 두 단계로 분리한다.
  1. 새 컬럼 추가 + 코드에서 구/신 컬럼 양쪽 호환
  2. 다음 릴리스에서 구 컬럼 제거
- **타입 변경**: 신 컬럼 추가 → 데이터 마이그레이션 → 구 컬럼 제거 순으로 분리.
- **스키마 변경과 기능 리팩터링을 하나의 커밋에 섞지 않는다.**

---

## 3. 트랜잭션

- 모든 `INSERT` / `UPDATE` / `DELETE`는 트랜잭션으로 묶고, 실패 시 `rollback()`을 명시적으로 호출한다.
- 여러 테이블에 걸친 삽입은 단일 트랜잭션으로 묶거나, 실패 시 이전 삽입을 롤백하는 보상 로직을 구현한다.
- `cursor.close()`는 `finally` 블록에서 호출한다.

---

## 4. 쿼리 로직 중앙화

**같은 조회 조건을 여러 곳에 중복 작성하지 않는다.**

- DB 접근은 Repository 계층에서 일괄 관리한다.
- 공통 필터·정렬 조건은 빌더 또는 헬퍼 함수 내부에서 관리한다.
- 검색 로직을 추가/수정할 때는 Repository 외부에 새 구현을 만들지 않는다.

---

## 5. SQL Injection 방지

- ORM 또는 파라미터 바인딩(`cursor.execute(query, (param,))`)만 사용한다.
- 사용자 입력을 쿼리 문자열에 직접 `format`하거나 concat하는 raw SQL 금지.

```python
# 금지
query = f"SELECT * FROM tbl_item WHERE id = '{item_id}'"

# 올바른 방법 — 바인딩 파라미터
query = "SELECT * FROM tbl_item WHERE id = %s"
cursor.execute(query, (item_id,))
```

---

## 6. 외부 인덱스·검색 엔진 사용 시 원칙

벡터스토어, 검색 엔진 등 외부 인덱스를 사용하는 경우에도 동일 원칙을 적용한다.

- **스키마(인덱스 구조, 차원, 매핑) 변경**이 필요할 때는 신규 인덱스 생성 → 데이터 이전 → 전환 순서로 점진적으로 진행한다. 기존 인덱스 삭제 후 재생성 방식은 **운영 환경에서 금지**한다.
- 인덱스 이름·컬렉션 이름은 코드에 하드코딩하지 않고 환경변수로 주입한다.
- 모든 저장소에서 동일한 문서 식별자(`doc_id`, `item_id` 등)를 공유하여 데이터 일관성을 보장한다.
- 중복 체크는 해시(SHA-256 등) 기반을 우선으로 하고, 벡터 유사도 기반 중복 체크는 보조 수단으로만 사용한다.

---

## 7. 디버그 로깅 환경 분리

- SQL 덤프·쿼리 로그는 `local` / `development` 환경에서만 활성화한다.
- 운영(`production`)에서는 `echo=False`이어야 한다.
- DB 커넥션 풀 로그는 `DEBUG` 레벨 이하로만 출력한다.

---

## 8. 수동 마이그레이션 관리

마이그레이션 도구 없이 수동 SQL을 사용하는 경우:

- DDL/DML은 `scripts/` 같은 전용 디렉토리에 추가한다.
- PR 본문에 적용 SQL과 롤백 SQL을 함께 첨부한다.
- 애플리케이션 부팅 시 자동 스키마 변경 코드는 **idempotent**하게 작성한다 — 컬럼/인덱스 존재 여부를 확인 후 없을 때만 추가.
