# CLAUDE.md — workspace_web 공통 안내

이 파일은 `workspace_web/` 아래 모든 하위 프로젝트에서 Claude Code가 공통으로 참고하는 워크스페이스 수준 안내서다.
(프로젝트별 세부 규칙은 각 프로젝트의 `CLAUDE.md` 와 `.claude/rules/` 를 따른다.)

---

## 공통 코딩 원칙 (Iron Rules)

최상위 `.claude/principles/` 는 스택과 무관한 **공통 Iron Rule 베이스**다 (보안·에러·완전성·코드스타일·프로젝트구조·search-before-build·agent-docs·api-conventions, 그리고 백엔드 전용 `database`·프론트 전용 `state-management`).
각 하위 워크스페이스의 `.claude/principles/<같은 파일명>` 은 동일 원칙을 그 스택의 경로·API·명령으로 구체화하는 레이어다.
충돌 시 **개념은 공통 베이스**를, **구체 예시는 프로젝트별 문서**를 따른다. 위 "프로젝트별 세부 규칙은 각 프로젝트의 `CLAUDE.md` 와 `.claude/rules/` 를 따른다"와 동일한 2단 상속 구조의 연장선이다.

## 작업 진행 방식 — Loop Engineering (조건부 적용)

**실질적 규모의 작업은 아래 루프를 따라 진행한다.** 사소한 작업은 적용하지 않고 바로 처리한다.

### 적용 판단 (먼저 이걸 정한다)

```
아래 중 하나라도 해당하면  →  루프 적용 (LOOP)
  - 변경 파일이 여러 개이거나, 신규 기능/엔드포인트/엔티티를 만든다
  - 도메인·API·DB 스키마·보안 경로·부수효과를 건드린다
  - 디버깅/리팩터로 영향 범위가 한 파일을 넘는다
  - 한 호흡에 끝나지 않고 여러 단계를 거쳐야 한다

모두 아니면  →  루프 생략 (바로 처리)
  - 단순 질문·설명, 오타/한 줄 수정, 포맷팅, 설정값 하나 변경
  - 읽기 전용 조회, 상태 확인
```

> 애매하면 적용하는 쪽으로 기운다. 단, "사소한데 의식만 루프로 포장"하지 않는다 — 생략이 맞으면 생략한다.

### 루프 적용 시 따르는 단계

상세 기준·체크리스트·안티패턴은 **`.claude/guides/loop-engineering/`** 에 있다. 해당 단계 문서를 펼쳐 그 기준으로 진행한다.

```
0. INTAKE      요구사항 → 측정 가능한 완료의 정의(DoD) 확정   → 00-intake.md
1. PLAN        DoD 를 작업 단위로 분해 + 의존성 정렬          → 01-plan.md
2. CONTEXT     기존 자산·규칙·영향 범위 탐색 (만들기 전 검색) → 02-context.md
3. ACT         한 단위만 실행 (TDD: RED→최소구현)            → 03-act.md
4. OBSERVE     실제 빌드·테스트 출력 수집 (추측 금지)         → 04-observe.md
5. VERIFY ★    다중 게이트 검증 + 재시도/에스컬레이션 판단    → 05-verify.md
6. CHECKPOINT  커밋 + 진행 상태 영속화 (재개 가능하게)        → 06-checkpoint.md
7. FINALIZE    통합 검증 + 증거 기반 보고                     → 07-finalize.md
```

핵심 안전장치(반드시 지킴): **종료 조건**(MAX_RETRY) · **검증 게이트**(실제 테스트로만 완료 판정) · **에스컬레이션**(막히면 정직하게 보고하고 멈춤) · **상태 영속화**(중단돼도 재개).

### 적용 방식

- 전체 단계를 기계적으로 나열하지 않는다. **그 작업에 필요한 단계만** 밟되, VERIFY(검증)와 FINALIZE(증거 보고)는 생략하지 않는다.
- 문서는 자동 주입되지 않으므로(`.claude/guides/`), 루프를 적용하기로 했으면 **해당 단계 문서를 그때 열어 읽는다.**
- 범용 문서다. 예시의 Gradle 명령은 해당 프로젝트 빌드 명령으로 치환한다.
